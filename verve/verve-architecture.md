# Verve — Production Architecture

Tagline: *Become a Stronger Communicator.*

---

## 1. Final Production Architecture

Four independently deployable services, one shared contract (OpenAPI schema generated by FastAPI, consumed by the frontend via a typed client).

```
┌─────────────────────┐         ┌──────────────────────┐
│   Next.js 15 (Vercel)│  HTTPS  │   FastAPI (Railway)  │
│  - App Router        │────────▶│  - REST API           │
│  - Server Components │◀────────│  - Background workers  │
│  - Client islands     │  JSON   │  - Auth (JWT+refresh) │
│    (recording, curtain)│        │  - Feedback engine     │
└──────────┬───────────┘         └─────────┬────────────┘
           │                                │
           │ signed upload URL              │ SQL
           ▼                                ▼
┌─────────────────────┐         ┌──────────────────────┐
│  S3-compatible store │         │   Neon Postgres      │
│  (audio files)        │        │   (all relational    │
└──────────┬───────────┘         │    data)              │
           │                     └──────────────────────┘
           │ file ref
           ▼
┌─────────────────────┐         ┌──────────────────────┐
│   Whisper API         │────────▶│   OpenAI API          │
│  (transcription)      │  text   │  (feedback generation)│
└─────────────────────┘         └──────────────────────┘
```

Design principle: **the frontend never talks to Whisper/OpenAI/storage directly.** It talks only to FastAPI. FastAPI orchestrates transcription + analysis as an async background job and the frontend polls (or receives a websocket push) for status. This keeps API keys server-side, keeps the client "dumb," and lets you swap Whisper/OpenAI for another provider later without touching the frontend.

Two repos (recommended over monorepo, since Vercel and Railway build independently and have different lifecycles):
- `verve-web` — Next.js
- `verve-api` — FastAPI

A shared `verve-contracts` package (or simply a generated `openapi.json` checked into `verve-web`) keeps types in sync — FastAPI auto-generates the OpenAPI spec, and `verve-web` runs `openapi-typescript` at build time to produce typed fetch clients. This removes an entire category of frontend/backend drift bugs.

## 3. Frontend Architecture

- **Routing**: App Router with route groups — `(marketing)` public, `(auth)` unauthenticated only, `(app)` protected. `middleware.ts` checks the access-token cookie and redirects unauthenticated users hitting `(app)` routes to `/login`.
- **Server vs. Client Components**: Landing page, rankings table, and progress page are server components fetching data at request time (fast, SEO-friendly, no loading spinners). Recording, curtain sequence, and any `useState`/`useEffect`-heavy interactive pieces are client components, isolated to the smallest possible leaf so the rest of the tree stays server-rendered.
- **Styling**: Tailwind config maps every hex in your design system to a semantic token (`bg-background`, `bg-surface`, `bg-card`, `bg-elevated`, `border-border`, `text-primary`, `text-secondary`, `text-muted`, `text-gold`, `text-gold-secondary`, `bg-burgundy`, etc.) so no raw hex ever appears in component code. shadcn/ui components are re-themed via these tokens, not overridden ad hoc.
- **Typography**: `next/font` loads Cormorant Garamond and Inter as variable fonts, exposed as `font-heading` / `font-body` Tailwind utilities.
- **Animation**: Framer Motion owns the curtain sequence and page-transition micro-interactions; CSS transitions handle hover states. Framer Motion is deliberately *not* used for the waveform (that's wavesurfer.js's job) to avoid two animation engines fighting over the same 60fps budget.
- **Component reuse fix from the audit**: `ModeCard`, `PromptCard`, and the radar chart become single parameterized components used by both the landing page teaser and the full training/progress pages — eliminating the duplication flagged in the audit.

## 4. Backend Architecture

Layered, dependency-injected FastAPI app:

`api (routes)` → `services (business logic)` → `models (ORM)` → `db (session/engine)`

- **Routes** only validate input (via Pydantic schemas) and call a service method. No business logic in route handlers.
- **Services** contain the actual logic (rating math, feedback generation, transcription orchestration) and are unit-testable independent of HTTP.
- **Workers**: audio processing is slow (upload + Whisper + OpenAI can take 10–40s) — never block an HTTP request on it. A background worker (Celery + Redis, or lighter-weight `arq`) picks up a `process_session_audio` job the moment a session's audio upload is confirmed. The API returns `202 Accepted` immediately with a `session.status = "processing"`; the frontend polls `GET /sessions/{id}` (or subscribes to a websocket channel) until `status = "complete"`.
- **Auth dependency** (`get_current_user`) is a single reusable FastAPI dependency injected into every protected route — no repeated auth-checking code.

## 6. API Route Design

```
POST   /auth/signup
POST   /auth/login
POST   /auth/refresh
POST   /auth/logout
POST   /auth/forgot-password
POST   /auth/reset-password

GET    /users/me
PATCH  /users/me

GET    /prompts?mode=&difficulty=
GET    /challenges/today

POST   /sessions                      # create session (mode, prep/speak time, prompt assignment)
GET    /sessions/{id}                 # poll status + result
POST   /sessions/{id}/upload-url      # returns signed S3/Cloudinary upload URL
POST   /sessions/{id}/complete        # client confirms upload finished → triggers worker job
GET    /sessions                      # user's session history (paginated)

GET    /feedback/{session_id}

GET    /rankings?mode=&period=
GET    /ratings/me?mode=              # history for progress chart

GET    /goals
POST   /goals
PATCH  /goals/{id}

GET    /dashboard/summary             # aggregated single call for the dashboard page
```

`GET /dashboard/summary` exists deliberately as a single aggregation endpoint so the dashboard doesn't fire 6 separate requests on load — it returns rating, streak, radar values, today's challenge, and a rankings preview in one payload.

## 7. Authentication Flow

- **Signup**: email + password → hash with `bcrypt`/`argon2` → create user row + empty `user_statistics` row → issue access token (15 min expiry) + refresh token (30 day expiry, stored as httpOnly, secure, sameSite=lax cookie).
- **Login**: verify hash → issue same token pair.
- **Access token**: short-lived JWT, sent as `Authorization: Bearer` header, contains `sub` (user id) and `exp`.
- **Refresh token**: httpOnly cookie only (never exposed to JS, mitigates XSS token theft), rotated on every use (old refresh token invalidated, new one issued) to mitigate replay.
- **Refresh endpoint**: frontend's typed API client wraps every request; on a `401`, it silently calls `/auth/refresh` once and retries the original request before surfacing an error to the UI.
- **Forgot/Reset password**: `/auth/forgot-password` issues a short-lived, single-use reset token emailed to the user; `/auth/reset-password` accepts `{token, new_password}`, validates expiry, invalidates the token after use.
- **Protected routes**: `middleware.ts` on the frontend checks for the access token cookie before rendering `(app)` routes; FastAPI's `get_current_user` dependency re-validates the JWT server-side on every request regardless (never trust the frontend gate alone).

## 8. Speech Processing Flow

```
1. Frontend: POST /sessions            → session row created, status=pending
2. Frontend: curtain animation + prep countdown (purely client-side timer)
3. Frontend: MediaRecorder captures audio; wavesurfer.js visualizes live input
4. Frontend: on stop → POST /sessions/{id}/upload-url → gets signed PUT URL
5. Frontend: PUT audio blob directly to S3/Cloudinary (bypasses API server for the heavy binary)
6. Frontend: POST /sessions/{id}/complete → status=uploaded
7. Backend worker picks up job:
     a. status=transcribing → call Whisper API with the stored audio URL
     b. status=analyzing    → call feedback_engine with {prompt, transcript, mode}
     c. persist feedback_reports row, update ratings + user_statistics
     d. status=complete
8. Frontend polls GET /sessions/{id} every ~2s (or subscribes via websocket)
   until status=complete, then navigates to /session/{id}/feedback
```

Uploading directly to storage (step 5) rather than routing the binary through FastAPI keeps the API server stateless and avoids memory pressure from large audio payloads on Railway.

## 9. Feedback Engine Design

Two-stage pipeline, both calls made server-side inside `feedback_engine.py`:

**Stage 1 — Structured scoring.** A tightly constrained prompt to the OpenAI API instructs the model to act as a communication coach and return *only* JSON matching a fixed schema:

```json
{
  "structure_score": 82,
  "relevance_score": 91,
  "evidence_score": 64,
  "persuasion_score": 76,
  "confidence_score": 80,
  "examples_score": 58,
  "strongest_skill": "Structure",
  "weakest_skill": "Examples",
  "next_focus": "Use one example for every major claim.",
  "annotations": [
    {"start": 0, "end": 12, "type": "strong_opening", "note": "Clear thesis stated immediately."}
  ]
}
```

Pydantic validates the response against this schema before it ever touches the database; a malformed response triggers one retry with a stricter reprompt, then falls back to a "processing delayed" state rather than showing broken data.

**Stage 2 — Tone enforcement.** The system prompt explicitly forbids AI/meta language ("as an AI," "based on the transcript") and mandates coach-register phrasing ("Your opening was strong — you stated your position in the first sentence," not "The model detects strong sentence structure"). This is what preserves the "not an AI product" feel from a backend that is, mechanically, an AI product — the presentation layer of the feedback is a design decision, not an accident.

`next_focus` is deliberately singular — one focus, not a list — matching the existing "Next Focus" UI element in the prototype.

## 10. Communication Rating System

ELO-derived, adapted for asymmetric practice (there's no live opponent — the "opponent" is an implicit performance bar).

**Expected performance** for a user at rating `R` against the mode's current average rating `R_avg`:

```
E = 1 / (1 + 10^((R_avg − R) / 400))
```

**Actual performance** `S` is derived from the session's composite score (mean of the six sub-scores, 0–100, normalized to 0–1):

```
S = mean(structure, relevance, evidence, persuasion, confidence, examples) / 100
```

**Rating update:**

```
R_new = R + K × (S − E)
```

- `K = 32` for a user's first 10 sessions (fast calibration), stepping down to `K = 16` afterward (stability) — same pattern chess ELO uses for provisional vs. established ratings.
- Rating is tracked **per mode** (Impromptu/Debate/Interview/Storytelling each have their own R) **and** as an overall composite (weighted average, recency-weighted so recent sessions count more) — this powers both mode-specific rankings and the single headline number on the dashboard.
- `R_avg` per mode is recalculated nightly from all users' current per-mode ratings, keeping the whole system self-balancing rather than pinned to an arbitrary constant.

## 11. State Management Plan

- **Server state** (user profile, sessions, rankings, feedback): **React Query**. Every API call goes through it — automatic caching, background refetch, and built-in polling (`refetchInterval`) for the session-processing status check.
- **Ephemeral client state** (in-progress recording, curtain sequence step, mic permission status, elapsed timer): **Zustand** store scoped to the session flow, reset on route change. This is intentionally *not* React Query, since it's not server data.
- **Form state**: React Hook Form + Zod for signup/login/reset-password/goal-creation forms — validation schemas live in `lib/validation/` and are shared between client-side form checks and (mirrored) backend Pydantic schemas so validation rules never drift between the two layers.
- **Auth state**: a thin Zustand `useAuthStore` holds the decoded user object client-side for UI purposes (name, avatar), but it is never the source of truth for authorization — that's always the httpOnly cookie + server-side JWT check.

## 12. Deployment Plan

- **Frontend → Vercel**: connect `verve-web` repo, auto-deploy on push to `main`, preview deployments per PR. Environment variables: `NEXT_PUBLIC_API_URL`, none of the actual secrets (those stay backend-only).
- **Backend → Railway**: Dockerized FastAPI + worker as two Railway services from the same image (`web` process runs `uvicorn`, `worker` process runs the Celery/arq worker) so they scale independently. Railway's built-in Redis add-on backs the job queue.
- **Database → Neon**: serverless Postgres, branch-per-environment (a `dev` branch for staging, `main` for production) so migrations can be tested against a copy of prod schema before promotion.
- **Migrations**: Alembic runs as a Railway release-phase command (`alembic upgrade head`) before the new app version receives traffic — never manually.
- **Storage**: S3-compatible bucket (Cloudflare R2 or AWS S3) with a lifecycle rule to auto-expire raw audio after N days once transcription is confirmed complete (reduces storage cost — you don't need to keep the audio forever, only the transcript + score).
- **Secrets**: OpenAI key, Whisper key, JWT signing secret, DB URL, storage credentials all live in Railway's environment variables, never in the repo.

## 13. MVP Milestones

1. **Skeleton** — Next.js + FastAPI both deployed and talking to each other (health-check round trip), auth working end-to-end (signup/login/protected route).

7. **Polish pass** — loading/empty/error states everywhere, accessibility audit, mobile recording QA (mic permission flows differ meaningfully on iOS Safari).

## 14. Technical Risks

| Risk | Mitigation |
|---|---|
| Whisper + OpenAI cost scales with usage; a viral moment could spike bills | Rate-limit sessions per user per day at the API layer; cache prompt-level feedback patterns where feasible |
| Mobile Safari's mic permission + `MediaRecorder` support is inconsistent | Feature-detect at session start; show a clear pre-flight permission screen before the curtain animation, not after |
| Long-running AI calls could leave sessions stuck in "analyzing" if a call fails silently | Worker jobs have a hard timeout + automatic retry (max 2) + a `failed` status the frontend can display with a "regenerate feedback" action |
| Rating volatility from small sample sizes early on | Provisional `K=32` window plus a "provisional" badge in the UI until 10 sessions are logged, matching chess rating conventions users already understand |
| Curtain/waveform animations janky on low-end devices | Framer Motion `will-change` + reduced-motion media query fallback; wavesurfer.js configured with a capped canvas resolution |
| Audio storage costs grow unbounded | Lifecycle rule auto-deletes raw audio after transcript is confirmed (transcript + scores are the long-term asset, not the audio) |
| Leaderboard computed live would be slow at scale | Materialized `leaderboards` table rebuilt on a schedule (e.g. every 15 min), not computed per-request |
| Tone drift — feedback engine starts sounding like "an AI" instead of "a coach" | Golden-set of example outputs used in prompt few-shot examples; regression-tested whenever the prompt changes |
