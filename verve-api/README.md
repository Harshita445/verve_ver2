# verve-api

FastAPI backend for Verve. Step 1 of the build order: auth (signup/login/JWT) round-tripping against a real Postgres database, verified end to end.

## What's implemented right now

- `GET /health` — service health check
- `POST /api/v1/auth/signup` — creates a user, returns a JWT access token
- `POST /api/v1/auth/login` — verifies credentials, returns a JWT access token
- `POST /api/v1/auth/logout` — stub (real invalidation lands with refresh tokens)
- `GET /api/v1/users/me` — protected route, requires `Authorization: Bearer <token>`

Passwords are hashed with `bcrypt` directly (not passlib — passlib 1.7.4's
bcrypt backend self-test is broken against bcrypt>=4.1, so we bypass it).

## Local setup

1. **Postgres**: create a database and user matching `.env`:
   ```sql
   CREATE USER verve WITH PASSWORD 'verve';
   CREATE DATABASE verve OWNER verve;
   ```
2. **Python env**:
   ```bash
   python3 -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   ```
3. **Environment**: copy `.env.example` to `.env` and fill in real values (at minimum, set a real `JWT_SECRET`).
4. **Migrate**:
   ```bash
   alembic upgrade head
   ```
5. **Run**:
   ```bash
   uvicorn app.main:app --reload
   ```
6. Visit `http://localhost:8000/docs` for interactive OpenAPI docs (auto-generated by FastAPI — this is also the source `verve-web` can generate a typed client from).

## Generating a new migration after model changes

```bash
alembic revision --autogenerate -m "describe the change"
alembic upgrade head
```
